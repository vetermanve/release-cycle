Инженерный подход

Частые релизы
релизные поезда

От больших релизов раз в месяц — к маленьким поставкам 2 раза в неделю и далее ежедневно. Всё через git и CI, с автоматической сборкой, конфликтами под контролем и чистыми ветками.

Листать: стрелки / пробел / колесо. Схемы — mermaid, примеры — из рабочего демо-полигона.

Релизные поезда01

01Зачем это всё

Было: большой релиз раз в месяц

  • Копятся изменения -> сложное тестирование
  • Высокий риск каждого релиза
  • Обратная связь — недели
  • Тяжело найти причину дефекта

Стало: маленькие частые поставки

  • Малый батч — меньше риск на изменение
  • Обратная связь — дни
  • Дефект ищется быстро (мало изменений)
  • Предсказуемость и прозрачность

Единица поставки — бизнес-требование (БТ). Релизы собираются в поезда по расписанию.

Зачем02

02Словарь

БТ

Бизнес-требование. Описание — в Jira. Реализация — в ветке feature/bt-<N>.

Релиз-поезд

Набор БТ на дату. Имя по дате 26.06.09 (ГГ.ММ.ДД — сортируется по времени).

Стенд

Окружение: dev, test, prepod, prod. На стенд едет ветка.

Стенд-ветки

dev-DATE, test-DATE, release-DATE — собираются CI, руками не трогаются.

Merge-ветка

merge/bt-X-bt-Y — карантин для разрешения конфликта между БТ. Об этом — вторая половина.

Релиз-репо

Источник правды: состав поездов (bt-set.yaml) и привязка стендов (stands/).

Источник правды07

07Стенды — единая политика деплоя

Привязка стенд -> поезд

# отдельный файл на стенд, внутри — дата поезда:
stands/dev      ->  26.06.09
stands/test     ->  26.06.09
stands/prepod   ->  (пусто = ничего)
stands/prod     ->  26.06.09

Один файл = один стенд (не общий yaml) -> нет гонок read-modify-write. Деплоится только привязанное. Откат = git revert коммита.

Команда = стенд

make dev DATE= BTS=
bt-set + привязать dev -> собрать dev (:8081)
make test DATE=
привязать test -> собрать test (:8082) НАПРЯМУЮ
make release DATE=
prepod (:8083) + prod (:8084); master НЕ трогаем
make accept DATE=
прод принят -> merge prod->master + tag -> рефреш dev/test

Авто-рефреш (пуш в feature / смена bt-set) -> только активные dev/test. prepod/prod — только явным промоушном. Любой стенд собирается напрямую, минуя dev.

Stop-the-line10
Часть вторая
Конфликты

Когда два БТ трогают один код. Главная цель: feature-ветки остаются чистыми — ни одного коммита друг друга.

Конфликты11

10Порядок мержа — отложенный merge

Ветка не вмержилась -> не валим поезд. Откладываем, мержим остальные, возвращаемся к ней позже. Крутим, пока есть прогресс.

flowchart TD
  ST[units: merge-ветки + все feature] --> P{проход по pending}
  P --> TRY[merge ветку]
  TRY -->|ок| KEEP[оставить в dev, progress=1]
  TRY -->|конфликт| AB[git merge --abort -> отложить]
  KEEP --> NEXT[следующая]
  AB --> NEXT
  NEXT --> MORE{остались pending?}
  MORE -->|да| P
  MORE -->|нет| DONE[готово]
  P -->|проход без прогресса| REAL[настоящий конфликт -> резолв]
  classDef d fill:#161b22,stroke:#30363d,color:#e6edf3

«Отложение после отложенного»: отложенная ветка на следующем проходе может вмержиться (её пара уже в dev) — или окончательно уйти в резолв.

Отложенный merge12

11Карантин резолва — merge-ветка

Идея

  • Нашли конфликтную пару (попарная проба master + A + B).
  • Создаём merge/bt-A-bt-B = master + обе ветки.
  • Конфликт разрешаем ТАМ, один раз, в карантине.
  • feature/bt-A и feature/bt-B — не трогаются.

Почему чисто: в каждой ветке могут быть 5 коммитов, конфликт — только в первых. Merge-ветка фиксирует резолв; новые коммиты едут обычным мержем feature-веток.

Карантин13

12Кто что делает при конфликте

CI (reconcile) ловит конфликт, находит пару, создаёт merge/bt-X-bt-Y (skeleton с маркерами), пушит, падает красным.
Разработчик чекаутит merge-ветку, разрешает маркеры, пушит. feature-ветки не трогает.
CI (rebuild) на повторе видит разрешённую merge-ветку, мержит её первой -> резолв становится предком -> feature-ветки добавляются чисто.
# демо: разрешить и пересобрать
make resolve REPO=svc-a MB=merge/bt-77-bt-78   # (в жизни — руками разраб)
make rebuild DATE=26.06.16                       # повторный reconcile

Используется только разрешённая merge-ветка (без маркеров <<<<<<<).

15Тройные и кластеры

reconcile авто-создаёт только парные skeleton. Кластер 3+ собирает разработчик — одной веткой со всеми БТ в имени:

Кластеры17

16Код из демо — отложенный merge

# reconcile.sh — итеративный отложенный merge
while :; do
  progress=0; still=()
  for u in "${pending[@]}"; do
    if git -C "$rdir" merge --no-edit --no-ff "origin/$u" >/dev/null 2>&1; then
      progress=1
      [[ "$u" == feature/bt-* ]] && merged_feats+=" ${u#feature/bt-}"
    else
      git -C "$rdir" merge --abort   # конфликт -> отложить
      still+=("$u")
    fi
  done
  pending=("${still[@]}")
  [ "$progress" = 1 ] || break       # проход без прогресса -> настоящий конфликт
done

17Код из демо — дискавери merge-веток

# reconcile.sh — берём все merge/* с БТ из поезда (>=2), + все feature
while read -r ref; do
  short="${ref#origin/}"
  bts="$(parse_bts "$short")"                 # bt-77-bt-78 -> 77 78
  [ "$(echo $bts|wc -w)" -ge 2 ] || continue
  okm=1; for b in $bts; do in_set "$b" "$ids" || okm=0; done
  [ "$okm" = 1 ] && merge_units+=("$short")    # БТ ветки все в поезде
done < <(git -C "$rdir" for-each-ref 'refs/remotes/origin/merge/*')

units=( "${merge_units[@]}" "${feat_units[@]}" )   # merge-ветки ПЕРВЫМИ
