# Частые релизы: техническая архитектура (углублённая)

**Статус:** обкатано на полигоне (см. `README.md`, `make demo`/`make check`).
**Отношение к манифесту:** этот документ уточняет и местами исправляет манифест и гайд разработчика. Где расходятся — приоритет за этим документом.

---

## 0. Суть модели (GitOps)

Desired state в git, CI приводит к нему. Три источника правды:

- **`feature/bt-<N>`** в микросервисах — реализация БТ.
- **`trains/<DATE>/bt-set.yaml`** в релиз-репо — состав поезда (список БТ).
- **`stands/<env>`** в релиз-репо — какой поезд на каком стенде (dev/test/prepod/prod).

На любое изменение CI **reconcile** пересобирает и деплоит **привязанные** стенды:
`<env>-DATE = master + БТ поезда + merge-ветки`. Симметрично для всех env — любой стенд можно
собрать напрямую (починить test минуя dev).

> Что эволюционировало от первых версий: ушли императивные bash-скрипты (`add-to-rc.sh`),
> ушёл «срез» `dev->test->release` (каждый env собирается независимо), ушло поле `migration`
> из bt-set (миграции по факту). Добавлены: единая политика деплоя, `release`/`accept` split,
> карантин конфликтов в merge-ветках.

---

## 1. Принципы

1. Единица поставки — БТ. Одна фича -> `feature/bt-<N>` в каждом затронутом микросервисе (имя по номеру БТ).
2. Микросервисы изолированы по репо. Multi-repo БТ -> одноимённые ветки во всех затронутых.
3. Разработка и тест параллельны, API-first.
4. Релизы собираются вокруг поездов по расписанию.
5. Декларативность: меняешь состояние в git -> CI приводит стенды к нему.
6. Stop-the-line: дефект останавливает поезд, испорченный поезд не реанимируется (раздел 7).
7. Деплоится только то, на что указывает `stands/<env>` (раздел 5).

---

## 2. Модель веток

В каждом микросервисном репозитории:

| Ветка | Назначение | Кто двигает |
|-------|------------|-------------|
| `master` | Истина прода | `accept` (merge `prod-<date>` после приёмки) |
| `feature/bt-<N>` | Работа над БТ | разработчик |
| `merge/bt-X-bt-Y[...]` | Карантин резолва конфликта между БТ | разработчик (раздел 6) |
| `<env>-<YY.MM.DD>` | Сборка поезда для env (`dev-`/`test-`/`prepod-`/`prod-`) | reconcile (пересобирает) |

Дата инвертирована (`ГГ.ММ.ДД`) — алфавитная сортировка = хронологическая.

**Ключевое:** `<env>-DATE` — НЕ срез предыдущего env, а **независимая сборка** от master с БТ поезда.
Поэтому dev/test/prepod/prod симметричны, и любой можно собрать без остальных.

```
                         master + БТ(поезда) + merge-ветки
feature/bt-N  ─┐
merge/bt-*    ─┼─ reconcile ─┬─ dev-DATE     (stands/dev=DATE)
bt-set        ─┘             ├─ test-DATE    (stands/test=DATE)
                             ├─ prepod-DATE  (stands/prepod=DATE)
                             └─ prod-DATE    (stands/prod=DATE)  ──accept──> master + tag
```

## 3. Релизный репозиторий — source of truth

```
release-repo/
├─ catalog/repos.yaml        # репо, которые reconcile сканирует (включая *-migrations)
├─ schedule.yaml             # каденс (Вт/Чт | daily), время отправления
├─ stands/                   # binding стенд -> поезд (файл на стенд, внутри — дата)
│  ├─ dev                    #   напр. "26.06.09" (пусто = ничего)
│  ├─ test
│  ├─ prepod
│  └─ prod
├─ trains/26.06.09/
│  ├─ bt-set.yaml            # DESIRED: состав БТ (правится через MR)
│  ├─ affected-repos.lock    # GENERATED: repo -> <env>-branch -> sha -> bts
│  ├─ release-page.md        # GENERATED: doc-as-code (описания БТ из Jira)
│  └─ postmortem.md          # GENERATED только при остановке
└─ .gitlab-ci.yml            # один job reconcile
```

### bt-set.yaml — просто список БТ

```yaml
train: "26.06.09"
status: open            # open | departed | shipped | stopped
bts:
  - id: 16
  - id: 25
```

Никакого поля `migration` — миграции детектятся по факту (раздел 6). `status`: `open` -> можно менять
состав; `stopped` -> reconcile пропускает; `shipped` -> принят в master (после `accept`).

### stands/<env> — binding (по файлу на стенд)

Файл на стенд (а не общий yaml) -> нет read-modify-write гонки. Содержит дату поезда или пусто.
Меняется через `make dev/test/release` (Commits API) или вручную в GitLab UI.

`catalog/repos.yaml` — список репо для auto-scan; новый сервис (и его `*-migrations`) добавляется сюда.

---

## 4. Reconcile — единый реконсайлер

Один CI-job (`.gitlab-ci.yml`), триггеры: push с изменением `trains/**/bt-set.yaml` или `stands/**`,
либо `ACTION=reconcile|accept` (multi-project trigger / явный вызов).

### Кто что рефрешит

| Триггер | Что рефрешит |
|---------|--------------|
| смена `stands/<env>` (явный промоушн `make dev/test/release`) | **именно этот** стенд (любой env) |
| смена `bt-set` (композиция) | только активные **dev/test** этого поезда |
| push в `feature/bt-*` (multi-project trigger) | только активные **dev/test** |
| `make accept` | merge `prod->master`+tag, затем рефреш активных **dev/test** |
| `make rebuild DATE=` | все привязанные стенды этого поезда |

**prepod/prod меняются ТОЛЬКО явным промоушном** — прод не дёргается пушем в фичу.

### Сборка одного env (идемпотентно)

1. `catalog/repos.yaml` -> репо. Для каждого: `git ls-remote` проверяет наличие `feature/bt-<N>` поезда —
   **незатронутые не клонируются** (оптимизация на масштабе).
2. Затронутый репо: клон, `git checkout -B <env>-DATE origin/master`.
3. Юниты = `merge/*` (БТ⊆поезда, первыми) + ВСЕ `feature/bt-<N>` (по возрастанию N).
4. **Отложенный merge** до фикспоинта (раздел 6).
5. Чисто -> push `<env>-DATE` (force), записать `affected-repos.lock`, deploy стенда.
6. `release-page.md` (БТ из Jira) -> коммит в релиз-репо (`[skip ci]`).

Идемпотентно: тот же `bt-set` (при неизменном master/ветках) -> то же дерево.

## 5. release / accept и политика деплоя

**Единая политика:** деплоится только то, на что указывает `stands/<env>`.

- **`make dev DATE= BTS=`** — пишет bt-set + `stands/dev` -> reconcile собирает dev.
- **`make test DATE=`** — `stands/test=DATE` -> собрать test (напрямую, можно минуя dev).
- **`make release DATE=`** — `stands/prepod+prod=DATE` -> собрать предпрод+прод. **master НЕ трогает.**
- **`make accept DATE=`** — прод принят (ничего не упало): `prod-DATE -> master` + tag `shipped-DATE`
  во всех затронутых репо, `status: shipped`; затем master подмержен -> активные **dev/test**
  пересобираются от свежего master (получают релиз как базу). **Только так релиз попадает в master.**

Почему split: иначе prod re-merge'ил бы master на каждом reconcile -> чужие поезда утекали бы в
параллельно собираемый test через master, а прод менялся бы от пуша в фичу.

### SLA (ориентиры процесса)

| Стадия | SLA |
|--------|-----|
| test (регресс + новое + contract) | <= 2 ч |
| предпрод (регресс) | <= 1.5 ч |
| прод | <= 30 мин |

Гейт — **scripted pass/fail** (без авторегресса SLA 2ч на daily недостижим). В полигоне гейт = внешний
сигнал (прошёл -> `make release`/`accept`; не прошёл -> `make stop`).

> **test-what-you-ship (риск deploy-слоя):** каждый env пересобирается из текущих веток, не пиннится.
> Держится дисциплиной «не менять ветки между test и prod» (или пин коммита/digest на стороне деплоя).

## 6. Конфликты: отложенный merge + карантин в merge-ветке

**Отложенный merge (порядок-независимость).** Конфликт -> `git merge --abort`, отложить, продолжить;
после прохода повторить отложенные (вдруг «пара» уже вмержилась); крутить, пока есть прогресс.
Проход без прогресса = оставшиеся реально конфликтуют.

**Карантин резолва (bt-ветки чистые).** При настоящем конфликте reconcile:
1. находит **пару** (попарная проба `master + A + B`);
2. создаёт `merge/bt-A-bt-B` (master + обе ветки) со skeleton-маркерами, пушит, pipeline падает;
3. разработчик чекаутит ветку, разрешает маркеры, пушит. `feature/bt-A` и `feature/bt-B` **не трогаются**.

**Сборка.** Юниты = ВСЕ `merge/*` (БТ⊆поезда, первыми) + ВСЕ `feature/bt-N`. Merge-ветка несёт резолв
(становится предком) -> feature-ветки добавляют новые неконфликтные коммиты без повторного конфликта.
Разные merge-ветки решают разные конфликты — включаются **все** (дизъюнктные композятся).
Неразрешённый skeleton, чьи БТ ⊆ разрешённой merge-ветки (эскалация пары в кластер), **пропускается**.

**Кластеры 3+** — вручную: `make mkmerge REPO= BTS=77,78,79` -> `merge/bt-77-bt-78-bt-79`
(подключается по имени, БТ⊆поезда). Эскалация пары в тройку = пересоздать тройную + удалить парную.

Правило: ветки `dev-/test-/prepod-/prod-` руками не правятся; резолв живёт только в `merge/*`.

---

## 7. Миграции (Liquibase) — по факту, не в манифесте

Каждый сервис `svc-X` имеет парный репо `svc-X-migrations` (Liquibase changelog) **в `catalog/repos.yaml`**.
Его `feature/bt-N` сканируется и собирается в `<env>-DATE` так же, как код — **по факту наличия ветки**.
Попал в сборку -> виден в `affected-repos.lock` / на стенде. Никакого поля в bt-set.

**Expand-contract — дисциплина changeset'ов** (не манифест): соседние версии кода (N и N-1) обе работают.
expand (добавить) — до кода; contract (удалить) — отдельным поздним поездом, когда старый код не в проде.
Деструктив и код-что-перестал-юзать — никогда в одном поезде (держит rollback безопасным).

---

## 8. Stop-the-line

Дефект на гейте:
1. `make stop DATE=` -> `status: stopped` + `postmortem.md` (doc-as-code). reconcile пропускает stopped.
2. Стенд-ветки не реанимируются. Дефектный БТ чинится в `feature/bt-N`.
3. БТ заезжает в следующий поезд (Вт/Чт -> четверг; daily -> следующий рабочий день + бездефектные БТ).

---

## 9. Rollback

- Откат стенда: `git revert` коммита `stands/<env>` -> reconcile собирает прошлый поезд.
- master растёт только через `accept`, поэтому откат стенда не задевает релизную базу.
- Схема БД совместима назад (expand-contract) -> откат кода безопасен без отката миграций.

---

## 10. Каденс и формат даты

- Старт **Вт/Чт**, цель — каждый рабочий день. `schedule.yaml` задаёт режим + время; `next-train` из него.
- Дата `ГГ.ММ.ДД` (инверсия) -> сортировка по алфавиту = по времени.
- Переход на daily: 3 цикла без остановок по дефектам после dev-теста; test->прод стабильно <= 4 ч;
  полная автоматизация; нет повторов на постмортемах.

---

## 11. Роли

- **Разработчик:** БТ в `feature/bt-N`; контракт с тестировщиком; добавление БТ в `bt-set.yaml` (MR);
  резолв конфликтов в `merge/*`.
- **Тестировщик:** тесты параллельно (с контракта); автрегресс+contract как гейт.
- **Релиз-инженер/DevOps:** промоушн (`stands/<env>`: dev/test/release/accept); reconcile в GitLab CI.
- **Product Owner:** приоритет БТ под каденс; ответственность за `bt-set.yaml`.

## 12. Решённые развилки (ADR-кратко)

| # | Решение | Почему |
|---|---------|--------|
| ADR-1 | Декларативный `bt-set.yaml` + reconcile | идемпотентность, аудит, GitOps |
| ADR-2 | `<env>-DATE` собирается с нуля от master (не срез) | симметрия env, сборка любого стенда напрямую |
| ADR-3 | binding `stands/<env>` (файл на стенд) | деплоится только привязанное; нет RMW-гонки |
| ADR-4 | Auto-scan `feature/bt-N` по каталогу + ls-remote фильтр | разработчику не вести список; не клонировать лишнее |
| ADR-5 | Единая политика: deploy по `stands/<env>`; авто-рефреш (feature/bt-set) только dev/test | прод стабилен, не дёргается пушем в фичу |
| ADR-6 | `release` без master; `accept` = merge prod->master+tag+рефреш dev/test | поезда не утекают через master; релиз попадает в master осознанно |
| ADR-7 | Миграции по факту (`*-migrations` в каталоге); expand-contract — дисциплина changeset'ов | без поля в манифесте; безопасный rollback |
| ADR-8 | Конфликт: отложенный merge + карантин в `merge/bt-X-bt-Y`; включаются все merge-ветки | bt-ветки чистые; разные конфликты — разные ветки |
| ADR-9 | Stop-the-line + postmortem-as-code, поезд не реанимируется | малый батч, чистый следующий поезд |

## 13. Открытые вопросы

- **Ссылка на образ в деплое (tag/digest)** — на стороне инфры; влияет на строгий test-what-you-ship.
- Feature flags как альтернатива строгой изоляции веток (отложено).
- Гранулярность стендов: общий vs ephemeral per-train.
- Производительность на сотнях репо: ls-remote фильтр уже есть; при росте — кэш/параллелизм клонов.
