# Частые релизы: техническая архитектура (углублённая)

**Статус:** черновик для обкатки на полигоне
**Отношение к манифесту:** этот документ уточняет и местами исправляет манифест и гайд разработчика. Где расходятся — приоритет за этим документом.

---

## Граница автоматизации (scope)

**Наша зона — git-оркестрация веток.** Система намерживает feature-ветки в стенд-ветки (`dev-/test-/release-`) во всех затронутых микросервисах и их migration-репозиториях, по составу поезда из `bt-set.yaml`.

- **Вход:** `feature/bt-<N>` ветки в репо + `bt-set.yaml` в релиз-репо.
- **Выход:** ветки `dev-/test-/release-<DATE>` в каждом репо + `release-page.md` + статус поезда.
- **Вне зоны:** как стенд-ветки деплоятся (ArgoCD/OpenShift), сборка образов, ссылки на образы (tag/digest), фактический rollout. Это существующая инфра, чёрный ящик. Мы только рождаем правильные ветки; деплой реагирует на них как сейчас.
- **Гейт** (тест прошёл/нет) — внешний сигнал от существующего CI/QA; мы его *потребляем* (cut `release-` или stop-the-line), но тесты не гоняем сами.

Всё, что про образы и rollout, помечено ниже как *вне скоупа* — фиксируем как риск для владельцев деплоя, но не реализуем.

---

## 0. Что изменилось относительно исходного манифеста

Исходные тексты задавали процесс императивными bash-скриптами (`add-to-rc.sh`, `start-train.sh`) и пересборкой ветки на каждом окружении. Исправления:

1. **Декларативная модель вместо скриптов.** Источник правды состава поезда — файл `bt-set.yaml` в релизном репозитории, а не запуск скрипта человеком. Меняешь набор БТ -> CI пересобирает стенд-ветку. Идемпотентность по построению.
2. **Идемпотентная пересборка вместо инкрементального merge.** Стенд-ветка каждый раз собирается заново от `master` + merge текущего набора БТ. Выдернуть БТ = убрать строку из `bt-set.yaml`. Никакого `un-merge`, никакого расхождения `master` на ежедневном каденсе.
3. **Миграции — отдельный репо/артефакт.** Liquibase-changelog каждого сервиса живёт отдельно и мержится в стенд-ветки параллельно сервису; дисциплина expand-contract — метаданные для деплоя (см. раздел 6).
4. **postmortem-as-code** при stop-the-line — отсутствовал в манифесте.

> **Вне скоупа (deploy-слой):** test-what-you-ship (reproducible build, digest-verify), ссылки на образы, фактический rollback. См. раздел "Граница автоматизации" и раздел 8. Фиксируем как риск, не реализуем.

## 1. Принципы

1. Единица поставки — бизнес-требование (БТ). Одна фича -> ветка `feature/bt-<N>` в каждом затронутом микросервисе. Имя по номеру БТ, не по таске.
2. Микросервисы изолированы по репозиториям. БТ на несколько сервисов -> одноимённые ветки во всех затронутых репо.
3. Разработка и тестирование параллельны, API-first: контракт первым, тестировщик берёт контракт в работу сразу.
4. Релизы собираются вокруг поездов по расписанию.
5. Жёсткие SLA на прогон поезда (раздел 5).
6. Автоматизация декларативная: меняешь состояние в релиз-репо -> CI приводит ветки и стенды к этому состоянию.
7. Stop-the-line: дефект на гейте останавливает поезд, испорченный поезд не реанимируется (раздел 7).

## 2. Модель веток

В каждом микросервисном репозитории:

| Ветка | Назначение | Кто двигает |
|-------|------------|-------------|
| `master` | Истина прода | merge `release-<date>` после прода |
| `feature/bt-<N>` | Работа над БТ | разработчик |
| `dev-<YY.MM.DD>` | Релиз-кандидат, авто-деплой на дев-стенд | reconciler (пересобирает) |
| `test-<YY.MM.DD>` | Срез на отправлении, авто-деплой на тест-стенд | train pipeline (cut от dev-) |
| `release-<YY.MM.DD>` | Срез после теста, авто-деплой предпрод -> прод | train pipeline (cut от test-) |

Дата инвертирована (`ГГ.ММ.ДД`) для алфавитной = хронологической сортировки.

Префикс `dev-`/`test-`/`release-` — это то, по чему существующая инфра катит ветку на соответствующий стенд. Поэтому релиз-кандидат называется `dev-<date>` (а не `rc-`): он сам уезжает на дев-стенд.

Поток:
```
feature/bt-N  --(reconcile)-->  dev-DATE  --(departure cut)-->  test-DATE  --(gate pass cut)-->  release-DATE  --(prod)-->  merge master + tag
   work                          dev-стенд                       тест-стенд                        предпрод->прод
```

## 3. Релизный репозиторий — source of truth

Один центральный репозиторий. Хранит desired state каждого поезда и сгенерированные артефакты.

```
release-repo/
├─ catalog/
│  └─ repos.yaml             # список микросервисных репо, которые reconciler сканирует
├─ schedule.yaml             # каденс (Вт/Чт | daily), время отправления
├─ trains/
│  └─ 26.06.09/
│     ├─ bt-set.yaml         # DESIRED: состав БТ поезда (правится людьми через MR)
│     ├─ affected-repos.lock # GENERATED: repo -> branch -> merge-sha
│     ├─ release-page.md     # GENERATED: doc-as-code страница релиза
│     ├─ gate-report.md      # GENERATED: результат гейта (pass/fail, потребляем извне)
│     └─ postmortem.md       # GENERATED только при остановке поезда
└─ .gitlab-ci.yml            # reconciler + train pipelines
```

### bt-set.yaml — единственный рукотворный файл состояния

```yaml
train: "26.06.09"
status: open            # open | departed | shipped | stopped
bts:
  - id: 16
    migration: expand   # none | expand | contract
  - id: 25
    migration: none
```

- `status` управляет жизненным циклом: `open` -> можно менять состав; `departed` -> заморожен (срез test- сделан); `shipped` -> в проде; `stopped` -> остановлен, состав уезжает в следующий поезд.
- `migration` на каждый БТ — фаза Liquibase (раздел 6). Метаданные для деплоя; в release-page, порядок применения исполняет инфра.

Каталог `catalog/repos.yaml` — список репозиториев, которые reconciler обходит при auto-scan:
```yaml
repos:
  - url: git@gitlab.local:svc/svc-a.git
  - url: git@gitlab.local:svc/svc-b.git
  - url: git@gitlab.local:svc/svc-c.git
```

Новый сервис -> добавить сюда. Без этого auto-scan его не видит (закрывает FAQ "мой сервис новый").

## 4. Реконсиляция ветки dev-DATE

Reconcile живёт в pipeline **релиз-трейна** (единственный владелец веток dev/test/release). Два триггера:

1. **Состав сменился** — push/merge в `trains/<DATE>/bt-set.yaml` (в релиз-репо) -> reconcile-pipeline.
2. **Контент фичи сменился** — push в `feature/bt-N` в сервисном репо. Сервисный CI на `feature/bt-*` по multi-project pipeline trigger (trigger-токен) дёргает pipeline релиз-трейна, передаёт номер БТ. Релиз-трейн находит открытый поезд с этим БТ (`status: open`) и пересобирает его `dev-DATE`. Сервисный CI сам в стенд-ветки не мержит — только сигналит.

Алгоритм (идемпотентный, для поезда `DATE` со `status: open`):
1. Прочитать `bt-set.yaml` -> список БТ.
2. **Auto-scan:** для каждого репо из `catalog/repos.yaml` проверить наличие `feature/bt-<N>` для каждого N из набора.
3. Для каждого затронутого репо **пересобрать `dev-DATE` с нуля**:
   - `git fetch`, `git checkout -B dev-DATE origin/master` (свежий срез master);
   - merge `feature/bt-<N>` в **детерминированном порядке — по возрастанию N** (порядок не зависит от порядка строк в файле, сборка воспроизводима);
   - `git push --force-with-lease origin dev-DATE` (ветка интеграционная, force допустим).
4. Записать `affected-repos.lock` (repo -> branch -> merge-sha) обратно в релиз-репо.
5. Push `dev-DATE` -> инфра катит на дев-стенд.
6. Сгенерировать `release-page.md` из `bt-set` + `affected-repos.lock`.

### Конфликты разрешаются в источнике, не в dev-DATE

Если merge `feature/bt-N` при пересборке конфликтует:
- pipeline **падает** (никакого авто-резолва — это была бы маскировка);
- конфликтный БТ помечается в `release-page.md` и в логе pipeline;
- разработчик чинит **в своей `feature/bt-N`**: merge свежего `master`, разрешает, push;
- reconciler перезапускается и пересобирает `dev-DATE` чисто.

Правило: ветки `dev-/test-/release-` руками не правятся никогда. Конфликт = сигнал чинить feature-ветку. Это заменяет раздел 4.2 исходного гайда (ручной merge в rc).

### Идемпотентность

Один и тот же `bt-set` (при неизменном `master`) -> одно и то же дерево `dev-DATE`. Пересборка с нуля исключает накопление состояния и делает "выдернуть БТ" тривиальным.

## 5. Отправление поезда и промоушн

Наша часть — только git-операции над ветками. Деплой и тесты — внешние; мы их *запускаем фактом появления ветки* и *потребляем их результат*.

### Отправление (cron, ориентир 10:00)

Train pipeline релиз-трейна для `DATE` со `status: open`:
1. Перевести `status: departed` (заморозить состав; новые БТ -> следующий поезд).
2. В каждом затронутом репо: `git checkout -B test-DATE dev-DATE` (срез без изменений), push.
   Появление `test-DATE` -> существующая инфра катит на тест-стенд (вне нашей зоны).

### Промоушн по сигналу гейта

Гейт (полный регресс + новая функциональность + contract) исполняет существующий CI/QA. Мы потребляем результат:
- **pass** -> в каждом репо `git checkout -B release-DATE test-DATE` (тот же commit), push -> инфра катит предпрод -> прод.
- **fail** -> stop-the-line (раздел 7).

После успешного прода: `merge release-DATE -> master` + тег во всех затронутых репо, `status: shipped`.

### SLA (ориентиры процесса, исполняются внешними стадиями)

| Стадия | Действие | SLA |
|--------|----------|-----|
| test-DATE | деплой тест-стенд + полный регресс + contract | <= 2 ч |
| release-DATE | деплой предпрод + регресс | <= 1.5 ч |
| прод | деплой прод | <= 30 мин |
| финал | merge `release-DATE` -> `master`, тег | — |

Итого test -> прод <= 4 ч (= критерий готовности к daily из манифеста). Гейт — **scripted pass/fail**, не ручная галочка: без автоматического регресса SLA в 2 часа на daily недостижим. На полигоне гейт моделируется внешним сигналом (pass/fail), его исполнение — забота владельца тестов.

> **Вне скоупа:** test-what-you-ship (reproducible build, digest-verify) при сборке из ветки. Если деплой ссылается на образы mutable-тегом, `release-`-пересборка может дать другой артефакт, чем тестировали на `test-`. Митигация — пин по digest или уникальному тегу на поезд — на стороне владельцев деплоя. Фиксируем как риск.

## 6. Миграции (Liquibase, параллельный репо/артефакт)

Каждый сервис `svc-X` имеет парный репо `svc-X-migrations` (Liquibase changelog). **Наша часть:** его feature-ветки `feature/bt-N` так же попадают в `catalog/repos.yaml` и так же мержатся в стенд-ветки `dev-/test-/release-DATE` параллельно сервису. Для git-оркестрации migration-репо — просто ещё один репо в наборе.

### Expand-contract — метаданные для деплоя (порядок применения вне нашей зоны)

Схема БД должна меняться так, чтобы соседние версии кода (N и N-1) обе работали:
- **expand** — добавить (колонку/таблицу/индекс), не ломая старый код. Применяется **до** кода, который этим пользуется.
- **contract** — удалить старое. **Отдельным, более поздним поездом**, когда старый код уже не в проде.

Поле `migration:` в `bt-set.yaml` объявляет фазу БТ. Это метаданные: мы их *записываем в release-page* и отдаём деплою. Фактический порядок применения (expand до кода, contract позже) исполняет существующая инфра — **вне нашей зоны**.

Правило-инвариант (для авторов БТ): деструктивная миграция и код, переставший использовать колонку, **никогда не в одном поезде**. Это держит rollback кода безопасным без отката схемы.

## 7. Stop-the-line и postmortem

Дефект на любом гейте (test или предпрод):
1. Поезд останавливается. Деплой дальше не идёт. `status: stopped`.
2. Ветки `test-DATE`/`release-DATE` заморожены, **не реанимируются**.
3. Авто-создаётся `trains/<DATE>/postmortem.md` (стаб: какие БТ, какой гейт, где не отсёк синхронный тест).
4. Дефектный БТ выпадает из набора. Чинится в `feature/bt-N`.
5. БТ заезжает в **следующий** поезд (добавлением в его `bt-set.yaml`):
   - режим Вт/Чт: следующий поезд — четверг;
   - режим daily: следующий поезд — следующий рабочий день; туда же все бездефектные БТ + фиксы, если готовы к утру.

Постмортем публикуется doc-as-code рядом с манифестом.

---

## 8. Rollback (deploy-слой, вне нашей зоны)

Откат прода исполняет существующая инфра, не наша git-оркестрация. Принципы фиксируем как требование к деплою:
1. Откат на предыдущий отгруженный поезд (по образу/манифесту, как умеет инфра).
2. Схема БД совместима назад (expand-contract, раздел 6) -> откат кода безопасен без отката миграций. Liquibase down на проде не применяем.
3. Инцидент -> postmortem-as-code.

Наш вклад в безопасный rollback — только дисциплина expand-contract в составе поездов; сам rollout не наш.

## 9. Каденс и формат даты

- Старт: **Вт/Чт**. Цель: **каждый рабочий день**.
- `schedule.yaml` задаёт режим и время отправления; `next-train` вычисляется из него.
- Формат даты ветки/папки: `ГГ.ММ.ДД` (инверсия) -> сортировка по алфавиту = по времени.
- Критерии перехода на daily (из манифеста, оставлены): 3 цикла без остановок по дефектам после dev-тестирования; test->прод стабильно <= 4 ч; полная автоматизация ветвления и доков; нет повторяющихся причин на постмортемах.

## 10. Роли (без изменений по сути)

- **Разработчик:** БТ в `feature/bt-N`; контракт с тестировщиком; добавление БТ в `bt-set.yaml` ближайшего поезда (MR в релиз-репо); починка конфликтов/дефектов в своей feature-ветке.
- **Тестировщик:** тесты параллельно с разработки (с контракта); автоматический регресс+contract как гейт; подтверждение на dev-стенде.
- **DevOps:** reconcile/train pipelines в GitLab CI (git-оркестрация веток); doc-as-code; статусы поезда. Деплой/сборка/rollback — существующая инфра, вне этой автоматизации.
- **Product Owner:** приоритет БТ под каденс; ответственность за состав `bt-set.yaml`.

## 11. Решённые развилки (ADR-кратко)

| # | Решение | Почему |
|---|---------|--------|
| ADR-1 | Декларативный `bt-set.yaml` + reconcile вместо bash add/remove | идемпотентность, аудит, GitOps |
| ADR-2 | Пересборка `dev-DATE` с нуля от master | нет un-merge, нет расхождения master на daily |
| ADR-3 | RC-ветка названа `dev-DATE` | префикс-деплой инфры катит её на дев-стенд |
| ADR-4 | Auto-scan `feature/bt-N` по `catalog/repos.yaml` | разработчику не вести список репо |
| ADR-5 | Деплой/сборка/rollback вне зоны автоматизации; test-what-you-ship — риск deploy-слоя | наша задача — только намерживать стенд-ветки |
| ADR-6 | Liquibase отдельным репо + expand-contract как метаданные | безопасный rollback, N/N-1 совместимость; порядок применения у деплоя |
| ADR-7 | Конфликт решается в feature-ветке, не в dev/test/release | не маскировать, не править интеграционную ветку руками |
| ADR-8 | Stop-the-line + postmortem-as-code, поезд не реанимируется | малый батч, чистый следующий поезд |

## 12. Открытые вопросы

- **Ссылка на образ в деплое (tag/digest)** — не решено. Deploy-слой; влияет на test-what-you-ship. Не наша зона, но фиксируем риск.
- Триггер пересборки от обновления `feature/bt-N`: multi-project pipeline trigger из сервисного CI в pipeline релиз-трейна (подтвердить на реальном GitLab).
- Нужны ли feature flags как альтернатива строгой изоляции веток (отложено).
- Гранулярность стендов: общий vs ephemeral per-train (полигон может проверить оба).
